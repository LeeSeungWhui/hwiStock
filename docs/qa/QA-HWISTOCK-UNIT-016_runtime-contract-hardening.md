# QA — Runtime Data And Execution Contract Hardening

## 1. Purpose

Prove that the operational paper-trading runtime has explicit, testable
contracts before any order-producing implementation row proceeds.

## 2. Scenario Matrix

| qa_id | priority | area | setup | expected_result | evidence |
| --- | --- | --- | --- | --- | --- |

## 3. Failure Conditions

This QA fails if any order-producing unit can proceed without:

- schema validation;
- finalized artifact manifest/hash;
- deterministic idempotency keys;
- executor lock/ledger;
- freshness TTLs;
- portfolio/order conflict reject codes;
- reservation accounting;
- paper-only broker guard; or
- ambiguous-submit reconciliation.
- expanded invalid fixture coverage for the P0/P1 Pro review findings.

## 4. Evidence Requirement

The closing evidence for this unit must include the contract document paths,
fixture names, validation outputs, and any accepted deferrals. Deferrals that can
lead to duplicate orders, live endpoint use, fake broker state, stale input
orders, or portfolio conflicts are blocking.
